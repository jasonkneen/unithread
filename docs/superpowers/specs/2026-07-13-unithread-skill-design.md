# unithread skill — design

**Date:** 2026-07-13
**Status:** approved (design), pending implementation plan

## Purpose

Package unithread as a portable Claude Code skill so an agent, asked to make an existing
codebase stop blocking its main thread, can find the offending work, move it onto real OS
threads, and **prove** it moved.

The skill exists because the library's value is easy to claim and hard to verify. An agent that
"offloads" a function, sees no errors, and reports success is the default failure mode. The
skill is built to make that outcome impossible to reach.

## Non-goals

- Not an npm package install path. unithread is unpublished; the skill vendors a file.
- Not a general performance skill. CPU-bound synchronous work only.
- Not a plugin (yet). A plugin is skills + commands in a repo; this upgrades into one later
  without rework.

## Packaging

Authored in this repo at `skill/`, installed by copy or symlink to `~/.claude/skills/unithread/`.
Authoring it in-repo means `assets/unithread.bundle.js` is regenerated by the same
`npm run bundle` that builds the library, so the vendored payload cannot drift from the source.

```
skill/
  SKILL.md                     trigger + 4-phase workflow + hard rules
  assets/unithread.bundle.js   13 kB, zero deps, runs unchanged in Node and browsers
  references/api.md            full API surface + gotchas
  scripts/verify-portable.mjs  proves a function survives the thread boundary
  scripts/measure.mjs          before/after harness
```

`SKILL.md` frontmatter carries `name: unithread` and a `description` that triggers on: janky UI,
blocked event loop, frozen page, "this loop is too slow", "run this in parallel", "use worker
threads", "offload this".

## Workflow

### Phase 1 — Audit

Find CPU-bound **synchronous** work on the main thread: pixel/matrix loops, parse or serialize
of large payloads, hashing, compression, recursive search.

**Instrument candidates and report real milliseconds. Never rank by guesswork.** The agent wraps
each candidate in a timer and runs it on representative input.

**Do not offload anything under ~16 ms.** Worker round-trip plus structured clone costs more than
the work. This rule rejects most bad offload ideas before they are written.

Present ranked candidates with measured costs. User picks.

### Phase 2 — Baseline

Measure the chosen candidate's cost **before** touching it, on a quiet machine.

`measure.mjs` must stop competing workloads before timing. This is not hypothetical: while
building the demo in this repo, a benchmark was run on a page that was itself running a CPU hog,
which made a healthy worker look 1.8x slower than the main thread and produced a false "the pool
is broken" conclusion. On a quiet machine the same worker measured 171 ms/job against the main
thread's 169 ms — no overhead at all.

Metrics:
- **Browser:** worst frame time and fps, sampled over a window. A frame is 16.7 ms.
- **Node:** event-loop block duration, plus wall clock for pool work.

### Phase 3 — Offload

1. Copy `assets/unithread.bundle.js` into the project (e.g. `src/lib/unithread.js`).
2. Rewrite the hot function to be **self-contained** (see Hard rules).
3. Pick the primitive:
   - `runInThread(fn, ...args)` — one-shot.
   - `Task.spawn(fn)` — persistent, called repeatedly.
   - `WorkerPool.create(fn, size)` + `pool.map(items, toArgs)` — parallel over a collection.
   - `spawnRemote(methods)` — an object of named methods as a service proxy.
4. Pass large buffers in the transfer list (moved, not copied — the source detaches).
5. Reach for `SharedArrayBuffer` only when state is genuinely shared. In browsers it requires
   COOP/COEP headers, which is a server change, not a code change.

### Phase 4 — Prove

Two gates. Both mandatory. Neither may be skipped because "tests pass".

**Gate 1 — portability.** Shipped functions cross the boundary via `Function.prototype.toString()`,
so a captured variable typechecks, lints, and passes review, then dies at runtime with a
`ReferenceError` inside the worker. `verify-portable.mjs` spawns the candidate in a real worker
with sample args and reports exactly which identifier leaked. Runtime check, not an AST parse: it
tests the actual failure rather than a model of it.

**Gate 2 — improvement.** Re-measure Phase 2's metric. Report before and after as numbers.
If the metric did not move, say so and revert. A no-op offload is never reported as a win.

## Hard rules (encoded in SKILL.md)

1. **Self-contained functions only.** No closure captures, no outer-scope imports. Everything
   arrives as an argument, a transferable, or shared memory.
2. **Nothing under ~16 ms gets offloaded.** The boundary costs more than the work.
3. **Workers are for CPU, not I/O.** Async I/O already yields; a thread buys nothing.
4. **Pool size defaults to `min(4, cores - 1)`.** Measured in this repo: a pool of 8 scored
   *worse* than a pool of 4 (2.0x vs 2.94x) — oversubscription.
5. **Queue depth 2 per worker.** At depth 1 each worker idles through the result round-trip:
   8.4 jobs/sec vs 12.4 at depth 2, measured.
6. **Measure on a quiet machine.** A busy one produces confident, wrong conclusions.
7. **"No errors" is not proof.** Only a moved number is proof.

## Error handling

- Worker errors propagate to the caller with message and stack (already true of the library).
- Portability gate failure → report the leaked identifier, fix by parameterising, re-run gate.
- Improvement gate failure → revert the change, report the measurement, do not claim success.
- No `SharedArrayBuffer` (missing COOP/COEP) → shared-memory features degrade with an explicit
  message; everything else (spawn, pools, proxies, transfers) still works.

## Testing

- The skill's two scripts are exercised against this repo's own demo as a fixture: a known-good
  offload (the `fib` workload) must pass both gates, and a deliberately closure-capturing function
  must fail Gate 1 with the correct identifier named.
- The library's existing suites remain the correctness backstop: Node 21/21, Chromium 11/11.
